# CAPÍTULO 1 – Introdução a Bancos de Dados e a Necessidade de Estrutura

Os Sistemas de Banco de Dados (BD) são um componente essencial da vida na sociedade moderna. Diversas tarefas cotidianas, como fazer um depósito bancário, reservar um voo ou realizar uma compra em um supermercado, envolvem a interação com um banco de dados. Sistemas de informações, mesmo os mais simples, necessitam armazenar dados para que possam recuperá-los e/ou alterá-los quando necessário.

## 1.1 Dados, Informação e Conhecimento.

Para compreendermos o universo dos bancos de dados, é fundamental distinguir entre dado, informação e conhecimento.

O **Dado** é o componente básico de um arquivo. Por dados, entende-se fatos conhecidos que podem ser armazenados e que possuem um significado implícito. Os dados são identificados em sua forma bruta e, por si sós, não levam à compreensão de um fato ou situação. Exemplos típicos de dados incluem nome, sobrenome, cidade ou bairro.

Um conjunto de dados que estão relacionados entre si é chamado de **Fato**. O **Metadado** é um dado sobre dado, fornecendo uma descrição das características do dado e do conjunto de relacionamentos que o ligam aos demais dados no BD.

A **Informação** é gerada após a interpretação ou o processamento dos dados. Ela é um agrupamento organizado de dados que faz sentido, gera conhecimento e auxilia na tomada de decisões em uma empresa. Por exemplo, o valor "2013" é um dado; ao combiná-lo com "Brasil", "1º Lugar" e "Copa das Confederações", obtém-se a informação de que "O Brasil foi campeão da Copa das Confederações do ano de 2013".

Um **Registro** é uma sequência de dados juntos que fornecem uma informação completa. Por exemplo, a ficha de um cliente contendo nome, CPF, RG e endereço.

O **Conhecimento** representa o discernimento obtido por meio de critérios e da apreciação dos dados e informações.

## 1.2 O Problema do Armazenamento: Da Planilha Simples (Dataset) e suas Falhas (Redundância) ao SGBD.

Antes do advento dos sistemas de banco de dados, as empresas mantinham as informações em arquivos físicos, como fichas de papel em arquivos de aço. Embora a migração para **arquivos digitais** (ou planilhas simples/datasets) tenha facilitado e agilizado o manuseio dos dados, essa **abordagem tradicional de processamento de arquivos** possuía graves falhas.

Nessa abordagem, cada usuário definia e implementava os arquivos necessários para o seu software, e os programas eram construídos para trabalhar com uma única base de dados específica. Isso levava a um forte acoplamento: qualquer mudança na estrutura de um arquivo exigia alterações em todos os programas que o acessavam.

A principal falha era a **redundância**. A redundância ocorre quando a mesma informação é armazenada em múltiplos locais. A abordagem de processamento de arquivos resultava em desperdício de armazenamento e duplicação de esforço para manter os dados atualizados.

A redundância acarreta três grandes problemas:
1. **Duplicação de Esforço:** Uma única atualização lógica precisava ser realizada várias vezes, uma para cada arquivo onde o dado estava gravado.
2. **Desperdício de Espaço:** O mesmo dado armazenado mais de uma vez consome espaço desnecessário, o que é crítico em bases de dados grandes.
3. **Inconsistência de Dados:** Arquivos que representam os mesmos dados podem se tornar inconsistentes se forem atualizados de forma desigual (por exemplo, se uma cópia for atualizada e outra for esquecida).

A **abordagem de Banco de Dados** resolve esses problemas ao utilizar um repositório único para manter os dados, os quais são definidos apenas uma vez. O objetivo ideal é ter um projeto que armazene cada item lógico de dado em apenas um lugar, o que é conhecido como **controle de redundância** ou **normalização de dados**.

## 1.3 Sistemas de Gerenciamento de Banco de Dados (SGBDs): Funções e Componentes.

Um **Sistema de Gerenciamento de Banco de Dados (SGBD)** é um sistema computadorizado ou conjunto de programas que permite ao usuário criar e manter um banco de dados. Trata-se de um software de propósito geral que facilita o processo de definir, construir, manipular e compartilhar bancos de dados entre vários usuários e aplicações.

O uso de um SGBD é essencial, pois ele retira da aplicação-cliente a complexidade de gerenciar o acesso, a manipulação e a organização dos dados. Além disso, proporciona suporte indispensável à medida que o volume de dados e o número de usuários crescem.

**Funções e Capacidades de um SGBD**

As principais funções de um SGBD incluem:
*   **Definição do BD:** Especificar os tipos de dados, estruturas e restrições.
*   **Construção do BD:** O processo de armazenar os dados em um meio controlado pelo SGBD.
*   **Manipulação do BD:** Inclui funções como consultar (recuperar dados), atualizar (alterar) e gerar relatórios. Um programa de aplicação acessa o BD enviando consultas (para recuperação) ou transações (que podem alterar os dados).
*   **Compartilhamento:** Permite que múltiplos usuários e programas acessem o BD simultaneamente.

Um sistema de banco de dados é composto pelo BD juntamente com seu sistema gerenciador. Os quatro componentes básicos de um Sistema de Banco de Dados são:
1. **Dados:** A coleção de registros armazenados.
2. **Software:** O SGBD, os programas aplicativos e os softwares de acesso aos dados.
3. **Hardware:** A máquina física, servidores ou dispositivos que processam e armazenam os dados.
4. **Usuários:** Incluindo o Administrador de Banco de Dados (DBA), Programadores de Aplicativos e Usuários Finais.

**Componente Essencial: Meta-dados e Catálogo**

Uma característica fundamental do SGBD é sua **natureza auto-descritiva**. Um sistema de banco de dados não contém apenas os dados, mas também uma definição completa da estrutura do BD e suas restrições, armazenada em um **catálogo de meta-dados**. O SGBD utiliza esses meta-dados para conhecer a estrutura dos arquivos, incluindo o tipo e o formato dos dados que serão acessados.

O SGBD também proporciona diversas vantagens, incluindo:
*   **Isolamento entre Programa e Dados:** Mudanças na estrutura de dados não exigem alterações na maioria das aplicações que os acessam.
*   **Suporte a Múltiplas Visões:** O SGBD permite que diferentes usuários vejam diferentes subconjuntos ou visões virtuais dos dados, conforme sua necessidade.
*   **Controle de Concorrência:** Implementa mecanismos para garantir que, quando múltiplos usuários tentam atualizar os mesmos dados, isso seja feito de forma controlada.
*   **Restrição ao Acesso Não-Autorizado:** Provisão de um subsistema de segurança e autorização para especificar restrições de acesso a dados confidenciais.
*   **Backup e Recuperação:** Mecanismos para recuperar de falhas de software ou hardware, restaurando o BD a um estado consistente anterior à falha.
*   **Impor Restrições de Integridade:** Capacidade de definir e aplicar regras de consistência, como tipos de dados, restrições de chave (unicidade) e integridade referencial (relacionamento entre registros em diferentes arquivos).

## 1.4 Modelos de dados e Arquitetura de Três Esquemas.

O **Modelo de Dados** é um conjunto de ferramentas que permite a descrição dos dados, sua semântica, relações e restrições de consistência. Ele é essencial para a definição da estrutura de um Banco de Dados.

Os modelos de dados são a base para a **Abstração dos Dados**, que é a capacidade do SGBD de esconder os detalhes da implantação da forma como os dados são armazenados e mantidos. Essa abstração é organizada através da **Arquitetura de Três Esquemas (ou Níveis de Abstração)**.

| Nível de Abstração | Descrição Principal | Modelo/Foco |
| :--- | :--- | :--- |
| **Nível Físico (ou Interno)** | Descreve **como os dados estão armazenados fisicamente** no computador. É o nível mais baixo de abstração, detalhando estruturas de armazenamento, tipos de campo e eficiência dos recursos. | **Projeto Físico** |
| **Nível Lógico (ou Conceitual)** | Descreve **quais dados estão armazenados** e **como eles se relacionam**. É a visão geral do BD, contendo a estrutura e os relacionamentos. É independente do SGBD específico. | **Modelo Conceitual** (e.g., Modelo Entidade-Relacionamento) |
| **Nível de Visão (ou Externo)** | Descreve apenas **partes dos dados**. É o nível mais próximo do usuário, mostrando apenas o que ele necessita e escondendo os detalhes da organização interna. | **Visões Específicas** |

O **Modelo Conceitual** (frequentemente usando o Modelo Entidade-Relacionamento) deve ser de alto nível, sendo de fácil entendimento para o usuário final. Ele descreve os objetos (entidades) e como eles se relacionam, omitindo detalhes de implementação. O **Projeto Lógico** transforma o modelo conceitual em um modelo mais próximo da implementação, como o Modelo Relacional (tabelas). Já o **Projeto Físico** é um refinamento do modelo lógico, dependente do SGBD escolhido (por exemplo, definindo tipos de dados como `VARCHAR(45)`).

## 1.5 ESTUDO DE CASO (E.C.): O Sistema de Biblioteca Escolar. (Prática relacionada ao conteúdo do capítulo no contexto do sistema da biblioteca.)

Para colocar em prática os conceitos fundamentais de estrutura e gerenciamento de dados apresentados, utilizaremos um contexto simples com o qual muitos estão familiarizados: o **Sistema de Biblioteca Escolar**.

Um sistema de biblioteca deve armazenar informações críticas para seu funcionamento, como:
*   Dados dos alunos e professores (Usuários Finais).
*   Dados dos livros/exemplares disponíveis (os Dados).
*   Registros de empréstimos e devoluções (Transações/Manipulação).
*   Definir regras (Restrições de Integridade) como o prazo de devolução ou a unicidade de códigos de livros.

A implementação deste sistema necessita de um SGBD para garantir que os dados sejam armazenados de forma coerente e recuperados com rapidez e segurança.
